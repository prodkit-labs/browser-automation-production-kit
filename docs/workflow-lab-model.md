# Workflow Lab Model

This repo treats each browser automation workflow as a small lab. A lab is more
than a runnable script: it includes fixtures, artifacts, metrics, failure
classification, cost controls, and a documented production handoff path.

## Lab Contract

Each workflow lab should document:

- problem
- when to use this workflow
- what it runs locally
- what it measures
- failure modes covered
- artifacts generated
- quickstart command
- benchmark output
- production upgrade path
- compliance notes
- provider handoff options

## Required Local Path

Every lab should keep a local or mock path runnable without paid services. This
path is used for development, CI, documentation screenshots, benchmark shape,
and regression tests.

## Evidence Layers

Use these evidence labels consistently:

- `measured`: generated by a reproducible script in this repo
- `estimated`: based on documented assumptions
- `not tested`: listed as a future evaluation candidate, not a recommendation

## Artifacts

Labs should preserve enough output to debug failures:

- HTML snapshots for failed pages
- screenshots for browser-backed failures
- request metadata and status codes
- error class and timeout settings
- raw benchmark CSV
- generated reports where relevant

Do not store secrets, raw credentials, private cookies, protected account
content, or private data in artifacts.

## Cost Model

Cost reports should distinguish:

- cost per 1k requests
- cost per 1k successful pages

Production decisions should prefer successful-page cost because failed requests
can still create provider, runtime, artifact, and debugging cost.

## Provider Handoff

Provider-backed paths must remain opt-in. Add them only when the lab already has:

- fixture scope
- raw benchmark output
- evidence label
- credential handling notes
- rate-limit assumptions
- tradeoffs
- nearby disclosure for any future public provider link

Do not turn provider handoff sections into rankings unless comparable benchmark
data exists.
