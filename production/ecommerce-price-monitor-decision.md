# E-Commerce Price Monitor Production Decision Notes

Use this note when a local e-commerce price monitor is ready to become a
scheduled production workflow.

This is not a provider ranking. Start with the local/open-source path, record
failure evidence, and compare paid provider paths only when the workload needs
them.

## Local Workflow

Run the fixture workflow:

```bash
python -m prodkit_browser.jobs.ecommerce_price_monitor \
  --fixture benchmarks/fixtures/ecommerce_pages.json
```

Expected outputs:

- price-change events
- selector drift report
- HTML artifacts
- summary JSON

## Production Questions

Before choosing a provider path, answer:

- How many product pages run per batch?
- How often does the batch run?
- Which pages require JavaScript rendering?
- Which failures are selector drift, timeout, 403/429, or parsing errors?
- Which failures should retry?
- Which artifacts must be retained for debugging?
- Does the workflow need region, session, or proxy behavior?
- What is the acceptable cost per 1k successful pages?

## Path Comparison

| Path | When it fits | Watch for | Evidence to collect |
|---|---|---|---|
| Local browser worker | Low to moderate volume, controlled targets, and manageable maintenance | host resources, browser dependencies, selector drift, retry growth | success rate, p95 latency, artifact size, failure classes |
| Proxy-backed browser worker | Local browser logic works, but network routing or region behavior matters | proxy cost, block rate, session handling, retry cost | block/429 rate, attempted pages, retry rate, cost per successful page |
| Scraping API | The team wants API-first extraction and less browser-worker maintenance | less browser control, rendering options, provider-specific response shape | request success, p95 latency, status codes, cost per successful page |
| Managed browser provider | Playwright/Puppeteer workflow needs hosted browser runtime, screenshots, traces, or sessions | browser minutes, session cost, artifact retention, concurrency limits | runtime cost, screenshots/traces, failure artifacts, p95 latency |

## Artifact Policy

Recommended first policy:

- retain full artifacts for failures
- sample successful artifacts
- keep price-change event JSON for every run
- keep selector drift reports until the selector is fixed
- avoid storing private or account-specific pages

## Retry Policy

Do retry:

- transient network errors
- provider throttling within a small retry budget
- browser startup failures

Do not blindly retry:

- selector drift
- parsing errors from known markup changes
- terms or access boundary failures
- repeated 403/429 spikes without reviewing workload and provider assumptions

## Provider Boundary

Specific provider names may appear only with evidence labels and public-safe
tradeoffs. Do not add provider links to this note unless there is nearby
disclosure and the open-source path remains documented.

Required evidence labels:

- `measured`: generated by benchmark scripts in this repo
- `estimated`: based on documented assumptions
- `not tested`: candidate only, not a recommendation
