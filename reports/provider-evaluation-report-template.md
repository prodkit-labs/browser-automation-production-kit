# Provider Evaluation Report Template

Use this report after a benchmark run has produced raw CSV. It is designed to
turn provider measurements into a readable production decision record without
creating a ranking page.

Local/open-source execution should stay documented before paid provider options.

## Disclosure Checklist

Complete this checklist before adding any public provider link:

- [ ] This report includes a local or open-source path.
- [ ] Any provider link appears near a disclosure note.
- [ ] Any affiliate relationship is disclosed before the link is clicked.
- [ ] Provider placement is based on workflow fit, evidence, and tradeoffs.
- [ ] No provider is described as universally best.
- [ ] No ranking is published without comparable benchmark data.
- [ ] Cost-control notes explain how to reduce spend.

## Evidence Freshness Labels

- `evidence`: broad evidence label used by older report rows.
- `evidence_status`: current run status, such as `measured`, `estimated`, or
  `not tested`.
- `measured_at`: when the raw benchmark row was generated. `not recorded` means
  the row came from an older CSV schema.
- `fixture_mode`: local fixture, local browser, hosted runtime, or candidate-only
  mode used for the row.
- `source_note`: short context for the evidence source. Treat candidate-only and
  not-tested rows as planning rows, not recommendations.

Use these evidence statuses:

- `measured`: generated by a reproducible benchmark script in this repo.
- `estimated`: based on documented pricing or clearly stated assumptions.
- `not tested`: included for future evaluation, not a recommendation.

Current evidence status: `<measured / estimated / not tested>`

## Category Tradeoffs

### Local/Open-Source Path

- Useful for parser development, fixture coverage, CI, and debugging without paid services.
- Watch for gaps between fixtures and live public pages.
- Upgrade only when production evidence shows reliability, scale, region, browser, or maintenance pressure.

### Hosted Automation Platform

- Useful for scheduled workflows, hosted runs, datasets, and operational convenience.
- Tradeoffs to document: platform runtime boundaries, storage model, workflow portability, scheduling semantics, and run cost.

### Managed Browser API

- Useful for Playwright/Puppeteer workflows that need hosted browser runtime, session support, screenshots, traces, or managed concurrency.
- Tradeoffs to document: cost, browser version support, debugging surface, concurrency limits, artifact retention, and provider-specific runtime behavior.

### Scraping API

- Useful for API-first public page extraction when teams want less browser worker maintenance.
- Tradeoffs to document: less browser-level control, rendering behavior, request limits, retry behavior, response shape, and cost per successful page.

### Proxy/Data Infrastructure

- Useful for region/session testing and high-volume public data infrastructure.
- Tradeoffs to document: terms review, operations work, block-rate measurement, bandwidth pricing, session behavior, and compliance boundaries.

### Deployment Platform

- Useful for self-hosted workers, scheduled jobs, and internal workflow tools.
- Tradeoffs to document: browser dependencies, worker resources, job scheduling, artifact storage, and maintenance effort.
